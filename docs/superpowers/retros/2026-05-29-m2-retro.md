# M2 Retro — 2026-05-29

**Sprint outcome:** Shipped. 114 backend tests green, frontend production build clean, all 5 batches landed and pushed to GitHub. Task 12 (real Stripe end-to-end smoke) deferred until user provides `STRIPE_SECRET_KEY` + `STRIPE_WEBHOOK_SECRET`.

## What worked

- **Spring application events + `@TransactionalEventListener` replaces Inngest cleanly.** Publisher writes to `payment_events`, listener fires after commit, idempotency check on `processed_at` gives replay safety. No external queue infrastructure.
- **DLQ pattern via a single table + `@Scheduled` drain** absorbs transient publish failures without changing webhook semantics. Drain extracted as `drainNow()` for synchronous testing.
- **`@MockitoBean StripeService`** kept the controller integration tests fast and deterministic; the real Stripe SDK only needs to exist on classpath for the type signatures. WireMock not needed at this level.
- **Commission disclosure UI reused across signup + listing-create.** Single `CommissionDisclosure` component; live fee preview hits `/api/providers/me/listings/preview-fee` which routes through `PricingService` — same boundary tests from M1 cover the math.
- **`requestsMaterialChange()` predicate on `EditRequest`** keeps the M2/M3 transition trivial: M2 implementation of `hasActiveBookings(...)` returns `false`; M3 swaps the implementation with no caller changes.

## What didn't (fold into M3)

### Bugs found during testing

1. **JPA defer-flush hides `DataIntegrityViolationException` from `@Transactional` controllers.** The unique-constraint violation only fires at commit time, past the catch block. Fixed in webhook controller via fast-path `findByStripeEventId` + `saveAndFlush`. M3 booking transaction has the same shape (`UNIQUE (listing_id, consumer_id) WHERE status IN ('pending','confirmed')`) — apply the same fast-path pattern.

2. **`@TransactionalEventListener(AFTER_COMMIT)` runs outside any transaction**, so listener-side repository writes need `@Transactional(REQUIRES_NEW)` explicitly. Easy to miss — only surfaced when adding `saveAndFlush` (which requires an active tx, unlike `save` which lazily opens one). M3 booking-confirmed handler needs the same annotation.

3. **`SPRING_SESSION.PRINCIPAL_NAME` is VARCHAR(100)** — Java records' default `toString()` includes every field and overflows for our `CurrentUser`. Fix: explicit `toString()` returning email only. Caught when admin tests started inserting sessions.

4. **LazyInitializationException on every DTO that walks associations** — same M1 pattern recurs. M3 booking DTOs will need either JOIN FETCH or explicit getter touches inside the transaction.

### Plan-vs-reality drift

- **Stripe SDK version drift.** Plan said `28.5.0`; Maven Central had `29.x` only. Same pattern as Spring Boot version drift in M1 — always query Central at plan-write time.
- **`@AuthenticationPrincipal CurrentUser`** doesn't bind when MockMvc uses `user("name").roles(...)` — that helper creates a `UserDetails` principal, not a `CurrentUser`. Have to use `authentication(new UsernamePasswordAuthenticationToken(currentUser, null, authorities))` instead.
- **`jsonb` round-trips reformat keys** (alphabetical, with spaces). Assertions on raw JSON strings need to be format-agnostic.
- **Spring Boot 3.5 returns 403 (not 401)** for both no-auth and wrong-role on `hasAuthority`-gated endpoints. Tests should assert `is4xxClientError()` if either is acceptable.

### Process notes

- **Test-pattern reuse.** `IntegrationTestBase` + `Factories` + the `authentication(currentUser)` helper made M2 test scaffolding fast. M3 will lean on these even more (concurrency tests, webhook simulation).
- **`sg docker -c '...'`** still required for every `mvn` invocation. Persistent for the session; user can drop it after re-logging into WSL or `newgrp docker`.

## Decisions for M3

| Decision | Choice |
|---|---|
| Booking transaction | `SELECT listing FOR UPDATE` + capacity check + INSERT, all in one `@Transactional` method. Stripe Checkout creation happens AFTER COMMIT (no network calls inside the row lock). |
| Idempotency on partial-unique violation | Fast-path `findExistingActive(listingId, consumerId)` before insert; treat the second-tap as resume-existing-pending. Matches the webhook idempotency pattern from M2. |
| Stripe Checkout `expires_at` | 30 minutes (Stripe's documented minimum); `pending_expires_at` = 35 minutes (the sweeper guard). |
| Webhook → confirm path | Same Spring event pattern as M2: `/api/webhooks/stripe` writes `payment_events` and publishes `StripeEventReceived`. New listener `BookingConfirmedHandler` flips status `pending → confirmed`, generates redemption code, emits `booking.confirmed` Spring event for the email scheduler. |
| Reminder + sweeper jobs | `@Scheduled` + injectable `Clock`; reminders via `step.sleepUntil`-equivalent — a simple "find bookings with start_time between now and now+1h and reminder_sent=false, send + mark" approach. |
| Refresh-status button on success page | Server-side rate limit: 1 call per 10 seconds per booking, max 6 per hour. |
| Tighten material-edit predicate | Swap M2's `return false` for `bookings.countByListingIdAndStatusIn(listingId, [pending, confirmed]) > 0`. |
| Tighten currency self-correct | Swap M2's "any listing" check for "any booking exists for any of my listings". |
